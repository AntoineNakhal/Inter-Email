Inter-Email V2 Documentation

Inter-Email V2 is an AI-assisted email workflow system focused on Gmail thread review, prioritization, structured analysis, and controlled draft support. The product has evolved from single-email processing to a thread-based model so the user can work the way real email conversations happen in practice.

Thread-based processing
Human-in-the-loop

Internal review UI
CRM-ready structured outputs

1. Product Goals

Operational value

Help a PO or non-technical end user quickly find the most important conversations without reading every single email.

Trustworthy assistance

Use AI to summarize, classify, and suggest actions while keeping manual review and decision-making in user control.

Future automation base

Structure outputs so the system can later integrate with CRM, tracking, and deeper workflow automation.

2. Solution Architecture

Main layers

  • Gmail fetch layer
  • Thread grouping and merge logic
  • AI analysis layer
  • Structured JSON persistence
  • Internal review application
  • End-user application
  • State persistence for user actions

Design principle

The system is intentionally human-supervised. AI produces summaries, categories, urgency, recommendations, and draft support, but the final operational judgment stays with the user.

4. User Interfaces

Internal review UI

  • Made for technical evaluation and debugging
  • Helps inspect thread grouping and AI decisions
  • Supports error analysis and trust tuning
  • Useful during refinement and validation

End-user UI

  • Made for daily operational use by a non-technical user
  • Shows recommended queue or all conversations
  • Supports thread detail view, Gmail links, and queue handling
  • Seen-state hides already-processed threads until they change

5. AI Decision Layer

Capability Purpose Operational value
Thread summary Summarize the whole conversation instead of isolated messages Gives fast context without reading the full thread
Urgency classification Identify which threads likely require faster attention Improves ranking and queue usefulness
Category classification Assign operational categories to each thread Supports triage and future reporting/CRM routing
Action guidance Suggest what should happen next Reduces interpretation effort for the user
Draft support Help generate response drafts when requested Speeds up response creation while keeping review manual

6. Draft Generation Flow

Desired user flow

  1. User opens thread details
  2. User clicks Create Draft
  3. Modal checks whether a date is needed
  4. Modal checks whether attachments are needed
  5. User adds keywords/instructions for AI
  6. System generates draft preview
  7. User reviews before any final action

Why this is better

  • Drafts are generated intentionally, not passively shown
  • The user sees only the inputs that matter for that thread
  • The flow feels guided and operationally safer
  • It better matches real-world drafting needs such as dates or files
Recommended AI metadata fields for this flow: needs_date, needs_attachment, and optional explanation fields to justify why the extra input is being requested.

7. Data Model and Persistence

Input data

Gmail messages, thread relationships, timestamps, senders, recipients, subject lines, and raw body content.

Processed objects

Thread-level entities containing merged conversation information, child messages, AI decisions, diagnostics, and version state.

User state

Seen-state persistence allows processed threads to stay hidden until a new message arrives or the thread version changes.

  • JSON-based structured output and persistence

9. Trust, Safety, and Quality

Safer merging

Thread merging now includes diagnostics and merge confidence to reduce incorrect conversation grouping.

Explicit AI fields

Decisions are more inspectable because the system exposes structured AI outputs instead of opaque results.

Review loop

The internal review UI and metrics layer make it possible to tune, validate, and improve the pipeline over time.

10. Current Status and Next Steps

Current phase

The system is in late refinement / early operationalization. It is already useful, but still needs validation around queue ranking, clarity of recommendations, and consistency of trust signals.

Best next priorities

  • Validate whether the recommended queue surfaces the right threads first
  • Improve explanation of why a thread is shown and what to do next
  • Complete the guided draft creation workflow
  • Document exact architecture and setup details
  • Prepare structured handoff toward CRM integration
