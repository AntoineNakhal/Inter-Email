Inter-Email V2 Documentation

Inter-Email V2 is an AI-assisted email workflow system focused on Gmail thread review, prioritization, structured analysis, and controlled draft support. The product has evolved from single-email processing to a thread-based model so the user can work the way real email conversations happen in practice.

1. Project Brief

This section summarizes the functional intent, operational goals, and delivery constraints of the Inter-Op email workflow project.

Project Context

Inter-Email V2 is an AI-assisted operational email workflow system designed to help a small organization process Gmail conversations more efficiently. The application is intended to reduce inbox overload, improve prioritization, and support human-in-the-loop decision-making through thread summaries, urgency classification, recommended actions, and guided draft generation.

Main Purpose

The purpose of the solution is not to replace human judgment, but to help the user quickly identify which conversations matter, understand them faster, and act on them with more consistency. The system is built for internal operational use and is designed to remain practical, trustworthy, and easy to use for non-technical users.

Business Objectives

  • Reduce the time required to review incoming email conversations.
  • Help the user focus first on the most relevant or urgent threads.
  • Support clearer operational follow-up and response planning.
  • Create structured outputs that can later support CRM integration.

User Objectives

  • Give the PO a simple daily queue of useful conversations.
  • Provide summaries without forcing full-thread reading every time.
  • Allow safe review before using any AI-generated draft response.
  • Keep the interface understandable for non-technical end users.

Technical Objectives

  • Process emails at the thread level instead of the single-message level.
  • Preserve child-message visibility for transparency and trust.
  • Store structured JSON outputs for review, ranking, and reuse.
  • Support future extension toward CRM and workflow automation.

Functional Requirements

  • The system must fetch Gmail messages and group them into threads.
  • The system must summarize each thread as a whole conversation.
  • The system must classify urgency, category, and suggested next action at thread level.
  • The system must expose child messages in the detail view.
  • The system must provide an internal review UI for technical validation.
  • The system must provide a cleaner end-user UI for daily use.
  • The user must be able to mark threads as seen.
  • Seen threads must stay hidden until the thread changes again.
  • The system must allow a guided draft creation flow instead of always showing a pre-generated response.
  • The draft flow must support conditional steps such as date input, attachment handling, keyword guidance, and final preview.

Non-Functional Requirements

  • The interface must remain simple, readable, and operationally clear.
  • The system must support human-in-the-loop review at all important stages.
  • The application must make AI decisions more transparent through explicit fields and diagnostics.
  • The queue behavior must remain predictable over time.
  • The system must be maintainable and structured for iterative improvement.
  • The solution should be lightweight enough for practical internal deployment and testing.
  • The architecture should be extensible toward future automation and CRM handoff.

Constraints

  • The project is being built in an internship context with limited weekly development time.
  • The solution must stay realistic for a small organization.
  • Human approval is required before sending or relying on generated outputs.
  • The system should prioritize usefulness and trust over aggressive automation.
  • The product must evolve incrementally, with validation before broader automation.

Acceptance Criteria

  • The system correctly groups Gmail messages into operational threads.
  • The recommended queue surfaces useful conversations in a meaningful order.
  • The end-user can open thread details reliably and understand what to do next.
  • The seen-state logic works consistently and only resurfaces changed threads.
  • The draft workflow supports guided generation instead of passive pre-drafting.
  • The internal review surface provides enough visibility to evaluate trust and errors.
  • The structured outputs are stable enough to support later CRM-related work.
This project is currently in a late refinement / early operationalization phase: the core workflow is usable, but ranking quality, trust clarity, and long-term operational usefulness still need validation before deeper automation.

6. Draft Generation Flow

Desired user flow

  1. User opens thread details
  2. User clicks Create Draft
  3. Modal checks whether a date is needed
  4. Modal checks whether attachments are needed
  5. User adds keywords/instructions for AI
  6. System generates draft preview
  7. User reviews before any final action

Why this is better

  • Drafts are generated intentionally, not passively shown
  • The user sees only the inputs that matter for that thread
  • The flow feels guided and operationally safer
  • It better matches real-world drafting needs such as dates or files
Recommended AI metadata fields for this flow: needs_date, needs_attachment, and optional explanation fields to justify why the extra input is being requested.

7. Data Model and Persistence

User state
