Inter-Email V3 — Architecture & Handoff

V3 is a modular monolith for AI-assisted email operations. A React frontend talks only to a FastAPI API, and the API delegates all real work to an internal backend/ package split into application services, domain models, provider adapters, and repositories. It is not microservices, but it already has clean internal boundaries — and a clear migration path to Postgres, background workers, and self-hosted AI without rewriting business logic.

Read this first. This document describes the V3 architecture as defined in V3/docs/architecture.md and as currently implemented on disk. As of April 2026 the system is essentially built end-to-end; see Section 8 — Current Build Status for the precise per-layer state and the remaining operational gaps (migrations, tests, type generation, sync durability). The V2 system this refactor extracted from is documented separately in inter_email_v_2_documentation.html.

1. Core Thesis

The one-minute pitch for any new senior dev joining V3.

What V3 is

One React frontend, one FastAPI backend service, one importable Python backend package, one repository-based persistence layer, one provider-agnostic AI abstraction, one local-first runtime data volume. Deployed as a single service today; ready to split into a worker process tomorrow without rewriting the business layer.

Why this shape

V2 proved the workflow but coupled UI, Gmail, AI, and persistence into the same scripts. V3 keeps the good parts — thread-centric processing, human-in-the-loop, structured AI outputs — and repackages them into deployment-ready shape: HTTP boundary, vendor-agnostic AI, SQL-backed state, environment-driven config.

Primary abstraction

The product unit is a thread, not a Gmail message. This is the right level for queueing, next-action recommendations, drafts, and CRM extraction. Gmail's threadId is the seed, but threads can merge across Gmail's split (HR/interview/calendar cases) via a domain-aware mapper.py.

One-minute summary

Gmail is normalized and grouped into product threads, threads are persisted and signature-hashed, then analyzed through a provider-agnostic AI layer that can route to OpenAI, Ollama, or a deterministic heuristic fallback. Every analysis also gets a verification pass for confidence and review routing.

Migration discipline

SQLite is the current persistence layer, but the application layer talks only to repositories. Switching DATABASE_URL to Postgres should be a config change, not a rewrite. Same principle for AI: business code calls the AIProvider interface, not OpenAI's SDK.

2. Layered Architecture

Six layers, strict dependency direction (top → bottom). The frontend imports no backend code. The API layer holds no business rules. Application services depend on repositories and providers, never on SQLAlchemy or vendor SDKs directly.

Frontend (React) API (FastAPI) Application services Domain (pure logic) Persistence / Repositories Providers (AI & Gmail) External systems

Hard rules — do not break

  • Frontend never imports backend code, never reads JSON files, never calls vendor SDKs.
  • API routers contain no business logic — only request/response mapping and dependency injection.
  • Application services never call OpenAI(), never write SQL, never touch googleapiclient directly.
  • Domain layer has no FastAPI, SQLAlchemy, or vendor imports.
  • Repositories are the only path to the database from the application layer.

Why it scales without rewrite

  • SQLite → Postgres: change DATABASE_URL; repositories shield the use cases.
  • OpenAI → Ollama: flip AI_DEFAULT_PROVIDER per task; business code unchanged.
  • BackgroundTasks → real worker: a worker process imports the same backend package and calls GmailSyncService directly.
  • One service → split deploy: the API and worker can ship as separate containers because they share only the package, not in-process state.

3. Refresh Gmail — End-to-End Sequence

The happy path when the user clicks Refresh Gmail in the inbox. Four real backend stages (fetch, persist, analyze, summarize), with live progress pushed into an in-memory store and polled by the frontend.

What makes this flow honest

  • Background, not synchronous. POST /sync returns 202 + run_id immediately; the frontend never holds the connection during a Gmail+AI run.
  • Email cleanup before AI. email_text.py strips reply chains, footers, and signature blocks so the AI never sees raw Gmail noise.
  • Cache by signature, not by ID. If the thread content didn't change, cached analysis is reused — even across regroupings.
  • Verification is persisted. accuracy_percent and needs_human_review live alongside the analysis row, not as a UI overlay.
  • Provider provenance is stored. Every analysis row records which provider/model produced it — important when comparing OpenAI vs. Ollama later.

What this flow is NOT yet

  • Not durable. Progress lives in sync_progress_store.py (in-memory). An API restart mid-sync loses progress; the run record is left in running.
  • Not incremental. Polling fetches a "recent window" — it does not use Gmail's history.list or push notifications, so changes outside the window are missed.
  • Not concurrent-safe across runs. Two overlapping Refresh clicks could spawn two BackgroundTasks against the same mailbox; protection is by-convention only.
  • Not isolated from process failure. If the AI provider hangs, the run hangs with it — no per-task timeout enforced at the orchestrator.

4. AI Provider Layer

The strongest design boundary in V3. Product code depends on the AIProvider interface, never on a vendor SDK. Provider routing is configuration-driven; fallback is per-task, not global.

Routing model

  • registry.py instantiates each implementation: heuristic, openai, ollama.
  • router.py resolves a provider per task using config and runtime state.
  • If runtime ai_mode == "local", every task routes to Ollama.
  • Otherwise, tasks route by AI_*_PROVIDER env vars, with heuristic as the safe deterministic fallback.
  • On provider error, the orchestrator falls back to provider_router.fallback_provider() — visible today in DraftService.

Task-specialized agents (not autonomous agents)

The local AI is not a tool-using autonomous agent. It is five task-specific prompt wrappers over one shared provider contract:

  • inbox_agent.py — analyze a thread (category, urgency, next action)
  • queue_agent.py — summarize the dashboard queue
  • draft_agent.py — guided draft generation
  • crm_agent.py — extract structured CRM fields
  • verification_agent.py — second-pass scoring

Interface shape (target)

class AIProvider(Protocol):
    name: str
    def analyze_thread(self, request: ThreadAnalysisRequest) -> ThreadAnalysisResult: ...
    def summarize_queue(self, request: QueueSummaryRequest) -> QueueSummaryResult: ...
    def draft_reply(self, request: DraftReplyRequest) -> DraftReplyResult: ...
    def extract_crm(self, request: CRMExtractionRequest) -> CRMExtractionResult: ...
    def verify_analysis(self, request: VerificationRequest) -> VerificationResult: ...

Both OpenAI and Ollama implementations normalize outputs after generation, so business code sees the same shape regardless of provider. That's what keeps the analysis service vendor-agnostic.

Senior-dev framing: the V2 "agent per task file" pattern is preserved by name, but intentionally reshaped — these are prompt constructors, not autonomous tool-using agents. That keeps the system testable and swappable, and avoids tying the architecture to any single SDK's agent concept.

5. Database Schema

SQLite today, Postgres-ready by repository contract. Eight core tables. Repositories are the only path in.

Table-by-table reference

TableOwnsKey columnsNotes
email_threads The product thread (post-grouping). external_thread_id, signature, account_email Signature is a content hash used for analysis cache and seen-state versioning.
thread_messages Individual Gmail messages, cleaned. external_message_id (global uniq) Global uniqueness lets a message move under a different thread on regrouping without dupes.
thread_analyses AI analysis results + verification. signature_at_analysis, provider, accuracy_percent One row per (thread, signature, provider). Stored fallback flag for evaluation.
thread_states Per-user view state. seen_signature_version Seen is versioned by signature — a changed thread re-surfaces.
review_decisions Internal review outcomes. decision, reviewer_email Append-only. Drives review queue and audit.
drafts AI-generated reply drafts. provider, model Latest-per-thread used by the UI; history retained for audit.
sync_runs Each Refresh Gmail invocation. status, stage, thread_count Source of truth for run progress; sync_progress_store is in-memory cache only.
runtime_settings Mutable runtime config. key / value (json) Holds ai_mode, active Gmail account, etc. Survives restart; user-editable from Settings.
Account isolation invariant. When the connected mailbox email changes, GmailConnectionService clears email_threads, thread_messages, thread_analyses, thread_states, and sync_runs for prior accounts. One account never sees another account's data. Worth a regression test in CI.

6. Frontend Shape

Route-driven React app. Owns presentation, local interaction, and query orchestration. Owns no business rules.

Routes

  • /inbox — InboxPage
  • /threads/:threadId — ThreadDetailPage
  • /review — ReviewPage
  • /settings — SettingsPage

Data layer

  • lib/client.ts — typed HTTP client (one instance, env-driven base URL).
  • hooks/useApi.ts — TanStack Query wrappers per endpoint.
  • useSyncMutation — POST /sync; useActiveSyncRun — poll progress.
  • Manual TS types in api.ts today; OpenAPI generation is the planned source of truth.

Progress UX

  • Source of truth: GET /sync/runs/active.
  • InboxPage applies client-side smoothing on top of polled values.
  • Active run polled every ~1s; idles otherwise.
  • Stage labels surface from the sync run record, not hardcoded in JSX.

7. Technology Stack

Picked deliberately for the migration path: keep Python where the AI/Gmail work is, give the UI a real frontend, and stay container-friendly from day one.

Frontend

  • React 18 + TypeScript
  • Vite
  • React Router
  • TanStack Query

API

  • FastAPI
  • Pydantic v2
  • Uvicorn
  • OpenAPI as shared contract

Backend & data

  • SQLAlchemy 2.x
  • Alembic (planned)
  • SQLite → Postgres
  • data/sqlite/app.db volume

AI & Gmail

  • OpenAI SDK (1.x)
  • Ollama HTTP (localhost)
  • google-api-python-client
  • google-auth-oauthlib

9. Strengths of the Architecture

What's worth defending in design review — even before the code catches up.

API-first by construction

The frontend imports zero backend code. That single rule eliminates most future deployment pain (separate origins, separate containers, separate languages) and makes the OpenAPI contract a real boundary, not a wishlist.

Provider-agnostic AI

The product layer never calls OpenAI(). Routing is per-task and config-driven, with a deterministic heuristic fallback that keeps the app responsive when providers fail. Migration to self-hosted AI is a config change, not a refactor.

Thread as the unit, not message

Queueing, next-action recommendations, drafts, and CRM extraction are all expressed against threads. Gmail's threadId seeds it, but the system can merge HR/interview/calendar threads that Gmail splits — which is the correct product behavior.

Persistence portability

Every read/write goes through a repository. Switching SQLite to Postgres should be a config flip plus a migration run — the application and domain layers stay untouched. The same shape leaves room for a future cache or object store.

Provenance is persisted

Every analysis row stores provider, model, prompt_version, and a fallback_used flag. That's not just nice-to-have — it's what makes A/B between OpenAI and Ollama a tractable evaluation problem instead of a guessing game.

Verification is first-class

accuracy_percent and needs_human_review are persisted alongside the analysis, not bolted on as a UI overlay. The review queue can be driven directly from the database without re-running the AI.

10. Risks & Push-back a Senior Dev Will Raise

The places the design will get challenged in review. None are blockers; all need owners.

AreaRiskWhy it mattersMitigation
Sync durability FastAPI BackgroundTasks + in-memory sync_progress_store. API restart loses in-flight progress; the run row is left running. Phase-1: pin progress to the sync_runs row. Phase-2: real worker (Arq, RQ, or Celery) reading from a queue.
Migration discipline create_all in database.py, schema evolution via repository ALTER TABLE. SQLite tolerates this; Postgres won't. Drift becomes silent. Adopt Alembic now, even before Postgres. Convert existing create_all into the baseline migration.
Type drift Frontend mirrors backend types by hand in api.ts while shared/openapi exists unused. API change without frontend update fails silently in TS instead of at compile time. Generate frontend types from the OpenAPI schema in the build (openapi-typescript). Fail CI on drift.
Gmail sync model Polling + recent-window. No history.list, no push. Changes outside the window are missed; cost grows linearly with mailbox size. Phase-2: switch to incremental history sync, persist historyId per account. Phase-3: Gmail push notifications via Pub/Sub.
Grouping heuristics mapper.py merges threads by subject/participants/links/HR signals. Better than raw Gmail threading, but false-positive/false-negative risk on edge cases. Keep merges traceable (store why two threads merged), and add a "split thread" admin action for recovery.
Stale runtime artifact backend/jobs/sync_runner.py imports services that don't exist. Confuses new devs into thinking it's the canonical entry point. Either delete it or refit it once the services land. Mark it explicitly non-canonical in the meantime.
Concurrency & timeouts Two refreshes can spawn overlapping background tasks; no per-task AI timeout in the orchestrator. Wedged provider call hangs the whole run; double-runs corrupt progress. Add a per-account "single active run" lock; enforce per-task timeout in the analysis service.
Account isolation regression Switching mailbox triggers a wipe of prior threads. Correctness-critical: one account must never see another's data. Pin a regression test on GmailConnectionService.switch_account; assert all related tables are scoped.
Local AI bypasses relevance gate In ai_mode=local, every fetched non-classified thread runs through AI. Intentional, but expensive on large mailboxes; could hide bugs in the relevance gate. Document explicitly; add a local_ai_max_threads safety knob in runtime_settings.

11. Next Refactors — Prioritized Backlog

With the architecture built, the remaining work is operational hardening. Order assumes a single dev or small team prioritizing correctness and deployability over new features.

  1. P0 — Test coverage. The single biggest correctness risk today. Add: repository tests (CRUD + signature/upsert behavior), service-level tests for GmailSyncService and ThreadAnalysisService (with in-memory SQLite + mocked providers), provider contract tests against the AIProvider interface (heuristic should pass the same tests as OpenAI/Ollama), and a frontend smoke test for the Refresh Gmail mutation.
  2. P0 — Alembic baseline migration. Replace create_all in core/database.py with an Alembic baseline. Convert any inline repository ALTER TABLE into versioned migrations. Required before any Postgres switch and before any deploy that mutates schema.
  3. P0 — Account-isolation regression test. Pin a test on GmailConnectionService.switch_account: assert that switching mailbox wipes email_threads, thread_messages, thread_analyses, thread_states, and sync_runs for the prior account. Correctness-critical and currently unverified.
  4. P1 — OpenAPI-generated frontend types. Add openapi-typescript to the frontend build. Generate types/api.ts from shared/openapi/openapi.json at build time. Fail CI on drift between the schema and committed types.
  5. P1 — Sync durability. Move the in-flight progress representation off sync_progress_store (in-memory) onto the sync_runs row. Add a per-account "single active run" lock; enforce a per-task AI timeout in ThreadAnalysisService. Restart-safe by construction.
  6. P1 — Resolve the legacy jobs/sync_runner.py. Either re-fit it against the current service signatures and keep it as a CLI utility, or delete it. The risk is a new dev assuming it's the canonical entry point.
  7. P2 — Worker process. Move the BackgroundTask invocation behind a worker (Arq is the lightest fit; Redis-backed, asyncio-native). Worker imports the same backend package, so no business-logic rewrite. Lets the API and worker scale and restart independently.
  8. P2 — Docker Compose & env templates. Add infra/compose/docker-compose.local.yml and docker-compose.selfhost.yml; commit a .env.example. Makes onboarding and self-hosted deployment a one-command path.
  9. P2 — Local AI safety knob. Add local_ai_max_threads to runtime_settings so that ai_mode=local cannot accidentally fan-out across a large mailbox.
  10. P3 — Provider evaluation harness. With provenance already persisted on thread_analyses, build a small Jupyter or CLI report: agreement rate between OpenAI and Ollama, confidence calibration, fallback rate. Now is the right time — the data model already supports it.
  11. P3 — Incremental Gmail sync. Switch from polling-window to users.history.list with per-account historyId persisted on the Gmail account record. Later: Gmail push notifications via Pub/Sub.
  12. P3 — Postgres switch. With Alembic in place, flip DATABASE_URL in self-host compose; verify SQLAlchemy types (especially JSON columns) port cleanly. Repository contracts should require zero change.
  13. P4 — Thread-merge transparency. Persist why two Gmail threads were merged by mapper.py (subject match, participant overlap, link match, HR signal). Add a "split thread" admin action so false-positive merges are recoverable in production.
